## 1. 概要

本アプリケーションは、ブラウザ上で音声ファイルまたは動画ファイルを読み込み、音声解析に応じてリアルタイムにビジュアル表現を行う **オーディオアナライザー / オーディオビジュアライザー** である。

本アプリは以下の2つの利用目的を持つ。

1. **再生モード**
   - 音声や動画を再生しながら、リアルタイムなアナライザー表現を楽しむ
   - 複数スロットを再生キューとして扱う

2. **録画モード**
   - アナライザーの描画を録画し、動画ファイルとして出力する

本アプリは **HTML / CSS / JavaScript を中心に構成し、特別なライブラリのインストールなしで簡単に起動できること** を最重要要件とする。

## 2. 開発方針

### 2.1 基本方針
- ブラウザのみで動作する構成を優先する
- 可能な限り **外部ライブラリなし** で実装する
- ユーザーに `npm install` 等を要求しない構成を目指す
- 実装の中核には Web 標準 API を利用する
  - 音声解析: `Web Audio API`
  - 描画: `Canvas 2D API`
  - 録画: `MediaRecorder API`
  - 再生: `HTMLAudioElement` / `HTMLVideoElement`

### 2.2 UI方針
- UI は簡素でよいが、**視認性** と **直感的操作性** を重視する
- ビジュアライザー表示領域を主役とする
- 背景は黒または白（UI切替可）固定とする
- コントロール部分は過度な装飾を避け、明快な構成にする

### 2.3 対象環境
- デスクトップブラウザを主対象とする
- 主な対象ブラウザ
  - Google Chrome
  - Microsoft Edge
- モバイルは Phase 13 でレイアウト面の対応を実施（狭幅画面での縦積みレイアウト・タッチターゲット拡大）。録画品質等の機能面はブラウザ実装依存のため引き続き保証対象外

## 3. 想定ユースケース

### 3.1 再生しながら鑑賞する
- ユーザーが音声ファイルまたは動画ファイルを読み込む
- 再生に合わせてビジュアライザーがリアルタイム表示される
- 再生中に色や動き、レイヤー構成を操作できる

### 3.2 複数ファイルを順番に流す
- 3つのファイルスロットにメディアを設定する
- 自動的に順番再生する
- 3番目の再生終了後は1番目に戻る

### 3.3 録画して書き出す
- 単一ファイルでアナライザーを再生する
- ビジュアライザー描画領域を録画する
- 録画結果を動画ファイルとして保存する

## 5. 用語定義

### 5.1 アナライザー
音声信号を解析して可視化する描画表現全般を指す。

### 5.2 レイヤー
音域ごと、または描画設定ごとに分けられた独立表示単位を指す。

### 5.3 0dB地点
可視化上の基準位置・基準辺・中心点を指す。  
厳密な音響計測値ではなく、表示上の「中立位置」の意味で用いる。

### 5.4 スロット
ユーザーがファイルを読み込むための枠。  
再生モードでは3つ存在する。

## 8. スロット・再生キュー仕様

※ 本節は初期設計で定義後、旧 Phase 4 で一旦実装中止となっていたが、Phase 12 で実装した。

### 8.1 スロット数
- 再生モードでは **3スロット**
- 録画モードでは **1スロットのみ有効**（自動循環は行わず、アクティブスロットのみを対象とする。録画中はスロット操作・次へ/前へを受け付けない）

### 8.2 再生順
- スロット1 → スロット2 → スロット3 → スロット1 → ...
- ループ循環を基本動作とする（設定済みスロットが1つだけの場合はそのスロットをループ再生する）

### 8.3 スロット未設定時の挙動
- 次スロットが未設定なら、その次の設定済みスロットを探す
- すべて未設定なら再生停止する

### 8.4 スロットUI表示項目
各スロットには以下を表示する。
- スロット番号
- ファイル名
- ファイル種別（音声 / 動画）
- 読み込み状態
- 再生対象状態
- 選択ボタン
- 削除ボタン

### 8.5 操作
- 再生
- 一時停止
- 停止
- 次へ
- 前へ
- 対象スロットの手動選択

## 9. 音声解析仕様

### 9.1 解析方式
- `Web Audio API` を利用する
- `AudioContext`
- `MediaElementAudioSourceNode`
- `AnalyserNode`

### 9.2 取得データ
- 周波数スペクトラムデータ
- 必要に応じて時系列波形データ

### 9.3 更新方式
- `requestAnimationFrame` を使用して描画同期を行う

### 9.4 基本表現
- 原則として **全音域を1つのアナライザーで表現** する

### 9.6 解析帯域
- アナライザーが対象とする帯域は **50Hz〜15kHz** とする
- 50Hz 未満の超低域ノイズ成分と、15kHz を超える超高域成分は除外する

### 9.5 拡張表現
- 音域を2〜4分割して複数レイヤー表示できる

## 11. アナライザー表示仕様

### 11.1 背景
- 背景色は **黒（`#000`）または白（`#fff`）をUIで切替可能**とする
- デフォルトは黒
- 残像フェードの色も背景色に連動する

### 11.2 アスペクト比
切替可能な表示比率:
- `16:9`
- `1:1`

### 11.3 アナライザータイプ
Phase 1・2 では汎用的な表示タイプを扱ってきたが、Phase 3 以降ではアナライザータイプを以下の2種類から明示的に選択可能とする。

- 棒グラフ
- 円形放射

Phase 6 では時間軸系・擬似3D系・流体/粒子系・幾何系の 12 タイプを追加し、計 14 タイプとする。詳細仕様は `doc/spec-phase6.md` に定める。

### 11.4 表現方法
各アナライザータイプに対して、以下の表現方法を選択可能とする。

- 棒
- 波形線
- 点

※ アナライザータイプと表現方法を分離して扱える構造を推奨する。  
※ Phase 3 以降では、対応可能な組み合わせを段階的に拡張できる設計とする。

### 11.5 0dB地点表現
0dB地点は、表示上の基準位置としてアナライザータイプごとに以下を採用する。

- 棒グラフ: 底辺基準
- 円形放射: 中心点基準

また、基準点からの表示位置を調整するため、**基準点オフセット** パラメーターを設ける。  
オフセットの最大値は **99** とする。

### 11.6 描画原則
- 再生音声に追従してリアルタイム更新
- 視認性を優先し、背景との十分なコントラストを保つ
- レイヤー数増加時にも破綻しない描画構造とする

### 11.7 棒グラフ時の表示モード（Phase 3 追加仕様）
棒グラフ選択時は、以下の表示モードを選択可能とする。

- 通常
- ミラー（上下対照）
- ミラー（左右対称）

### 11.8 残像表現（Phase 3 追加仕様）
アナライザーに残像表現機能を追加する。

- 最も弱い設定では残像の数は **1つ** で、減衰も早い。
- パラメーターを上げると、残像の数が増え、減衰時間も長くなる。
- 過剰設定では描画が飽和しうるため、以下の方針を採用する。
  - 残像数の上限（最大値）を実装で制御する。
  - 描画飽和が発生しやすい高値域では、ユーザーに注意喚起するか上限でクランプする。
  - 残像の合成は既存描画との輝度加算を基本とし、白飛びが目立つ場合は乗算やアルファ合成も検討する。

## 13. 再生モード仕様

### 13.1 概要
- 3スロットキューを利用する通常鑑賞モード

### 13.2 動作
- 選択されたスロットのメディアを再生
- 音声解析を行いビジュアライザーを描画
- 終了時は次スロットへ自動遷移

### 13.3 動画ファイル利用時
- 動画の音声成分を解析対象とする
- Phase 10 以降、動画映像をビジュアライザーの背景として合成表示できる（§14.9）
- 合成を使わない場合、映像そのものは表示しなくてよい

## 14. 録画モード仕様

### 14.1 概要
- 単一ファイルを再生しながらビジュアライザー表示を録画する

### 14.2 利用制約
- 利用可能スロットは1つのみ
- キュー再生は無効

### 14.3 録画対象
- ビジュアライザー描画領域のみを録画対象とする
- UIパネルは録画対象外とする

### 14.4 出力形式
- `MediaRecorder.isTypeSupported()` で対応可能な形式を自動選択する
- 優先順位: `video/mp4`（H.264/AAC の一般的プロファイル候補）→ `video/webm`（VP9/VP8）
- Chrome 130+ / Safari では MP4、Firefox など非対応ブラウザでは WebM にフォールバック
- 保存ファイル名の拡張子（`.mp4` / `.webm`）は実際に録画した MIME タイプから自動決定する
- エンコードビットレートは解像度・フレームレート・画質プリセットから自動算出する
  - 画質プリセット（Phase 11）: 低 / 標準 / 高 = 0.08 / 0.15 / 0.25 bit/pixel/frame
  - 標準時の下限 6 Mbps・上限 24 Mbps はプリセット係数に比例してスケールする（低画質時に下限クリップで実質無効化されるのを防ぐ）
  - 音声: 192 kbps（プリセットに依らず固定）
- WebM 出力時は録画終了後に `Segment > Info` へ `Duration` 要素を書き込み、再生時間情報を持つファイルとして保存する（MediaRecorder の WebM は Duration を持たず、編集ソフトで長さ不明になる問題への対策。外部ライブラリは使用しない）
- 録画データは 1 秒間隔のタイムスライスで回収し、長時間録画時のチャンク肥大とデータ損失リスクを抑える

### 14.5 操作
- FPS選択（25 / 29.97 / 30）
- 録画開始
- 録画停止
- 保存
- 再録画

### 14.6 保存ファイル名
- 初期仕様では日時ベースの自動命名でよい
- 将来的に任意名称指定を検討する

### 14.7 A/V同期
- 録画開始時は以下の順序で処理し、録画冒頭の映像と音声のずれを防止する
  1. `AudioContext` の `resume()` 完了を待つ（suspended のままでは音声トラックにデータが流れないため）
  2. `MediaRecorder` の `start` イベント（キャプチャ開始）を待つ
  3. その後にメディア再生を開始する
- 録画終了時の停止指示時刻から実測した録画長を WebM の `Duration` に使用する

### 14.8 オフライン書き出し（Phase 7）
- 通常の録画（14.1〜14.7）とは独立した機能として、音楽/動画ファイルを**再生を伴わず**解析し、現在のビジュアライザー設定に合わせて動画ファイルへ書き出す
- 対応ブラウザ: Chrome / Edge（`OfflineAudioContext` / WebCodecs API に対応する環境）
- 処理方式
  1. ファイルを `AudioContext.decodeAudioData()` で `AudioBuffer` にデコードする
  2. `OfflineAudioContext` 上で各出力フレーム時刻の周波数/時間波形スナップショットを決定的に採取する（実時間より高速に処理される）。解析経路は 14.8.3 を参照
  3. 採取したフレーム列を、既存のレンダラー群を用いて固定 dt（1/FPS）で1枚ずつ描画する
  4. `VideoEncoder` / `AudioEncoder` でエンコードし、専用のコンテナマクサーでコンテナ化する（コーデック/コンテナ選定は 14.8.1 を参照）
- 出力仕様
  - 映像: 16:9 選択時 1920×1080、1:1 選択時 1080×1080
  - FPS: 25 / 29.97 / 30 から選択（録画モードと同じ選択肢）
  - ビットレート算出式は録画モード（14.4）と同じ（画質プリセット 低/標準/高、標準時 0.15 bit/pixel/frame・6〜24 Mbps、音声 192 kbps）。画質は録画モードとは独立に選択できる
  - シーク索引（WebM: Cues / MP4: `mfra`）を書き込む。録画モードの Duration 後付けパッチのみの WebM より編集ソフトでの扱いやすさが高い
- 操作: 音楽ファイル選択（録画モードの再生スロットとは独立）/ FPS選択 / 書き出し開始 / キャンセル / 保存
- 書き出し開始時点の設定をスナップショットとして使用し、進行中に UI 側の設定を変更しても書き出し中の内容には影響しない
- 外部ライブラリなしの方針を維持する（コンテナ書き込みも自前実装）

### 14.8.1 コンテナ・コーデック選定（Phase 9.1）
- MP4（H.264/AVC + AAC）を優先し、非対応環境では WebM（VP9→VP8 フォールバック + Opus）を使用する
- 候補は `VideoEncoder.isConfigSupported()` を用いて上から順に判定し、最初に対応した組み合わせを採用する
  1. `avc1.640028`（High profile）
  2. `avc1.4d401f`（Main profile）
  3. `avc1.42e01e`（Baseline profile）
  4. `vp09.00.10.08`（WebM フォールバック）
  5. `vp8`（WebM フォールバック）
- MP4 コンテナは自前実装の fragmented MP4（fMP4）マクサー（`js/mp4-muxer.js`）で生成する
  - `ftyp` / `moov`（`mvhd` / `trak` × 映像・音声 / `mvex`）/ 映像キーフレームごとに区切られた `moof`+`mdat` の断片群 / `mfra`（シーク索引）で構成する
  - コーデック設定情報（`avcC` の AVCDecoderConfigurationRecord、`esds` の AudioSpecificConfig）は手動でビットストリームを解析せず、WebCodecs の `EncodedVideoChunkMetadata.decoderConfig.description` から直接取得する
  - 29.97fps 選択時は timescale を 30000、サンプル長を 1001 として正確に表現する（他 FPS は timescale=FPS、サンプル長=1）
- ファイル名の拡張子は実際に選定されたコンテナ（`video/mp4` または `video/webm`）に応じて自動的に決定する

### 14.8.2 オフライン書き出しでの動画合成（Phase 10.2 / 14.1）
- 書き出し対象に動画ファイルを選択し、かつ動画合成（§14.9）が有効な場合、その動画の映像フレームを各出力フレームの背景として合成する
- 処理方式は2経路を持ち、自動選択する
  1. **デコーダー方式（Phase 14.1・優先）**: WebM（VP8/VP9）は自前デマルチプレクサ（`js/webm-demuxer.js`）で映像チャンク列を取り出し、WebCodecs `VideoDecoder` で順次デコードする。実時間シーク待ちが無いため高速。`MediaRecorder` のストリーミング出力（Segment/Cluster が不定長）にも対応する
  2. **シーク方式（Phase 10.2・フォールバック）**: オフスクリーンの `<video>` 要素を各解析フレーム時刻へ `currentTime` でシークし、`seeked` イベントでフレーム到着を待ってから `drawImage` する。MP4 等の非WebMファイル・非対応コーデック・`VideoDecoder` 非対応環境・デコード途中の失敗時に使用する
- 描画仕様（cover フィット・不透明度・合成モード・selfClear タイプは対象外）はライブ表示（§14.9）と同一
- 頑健性: 映像トラックを取得できないファイルや、シークできないフレームがあっても書き出し全体は停止せず、該当フレームの合成のみスキップして続行する
- UI: 動画合成セクションは、ライブ再生用の動画読込中に加えて、オフライン書き出し対象が動画ファイルの場合にも表示する。どちらの対象もなくなった場合は設定を自動オフにする

### 14.8.3 解析経路（Phase 9.2）
- 解析（スナップショット採取）は AudioWorklet ベースを優先し、非対応環境では従来の `ScriptProcessorNode` + `AnalyserNode` へフォールバックする
- `AnalyserNode` はメインスレッド専用でワークレット（別レルム）から利用できないため、ワークレット経路では Web Audio API 仕様の `AnalyserNode` と同じ手順（Blackman窓 α=0.16 → FFT 1/N 正規化 → 線形振幅の時間平滑化EMA → dB変換 → minDecibels/maxDecibels による byte マッピング）を自前実装する（`js/fft.js` の `SpectrumAnalyzer`）
- ワークレットモジュール（`js/analysis-worklet.js` が生成）は `file://` 直開きでも動作するよう data: URL として登録する（blob: URL は `file://` オリジンで `addModule` が拒否されるため使用しない）
- 両経路はスナップショット粒度（2048サンプル境界）・平滑化の進み方・モノラルダウンミックス規則を揃えており、同一音源に対して実用上同一の出力を返す（実測: 時間波形バイト列は完全一致、周波数バイト列は誤差±1がごく僅か）
- どちらの経路が使われてもユーザーから見える動作・出力仕様は変わらない

### 14.9 動画合成表示（Phase 10.1）
- 動画ファイル読込時のみ、動画の映像フレームをビジュアライザーの背景として合成表示できる
- 合成方式: 動画フレームを `drawImage` でキャンバスへ cover フィット（アスペクト比が異なる場合は中央基準でトリミング）で描画し、その上にアナライザーのレイヤーを重ねる
- パラメーター: 有効/無効・不透明度（0〜100%）・合成モード（通常/加算/乗算/スクリーン）
- 全面を自分で塗り直すアナライザータイプ（スペクトログラム等の selfClear タイプ）では、動画合成は描画されない（アナライザー自体が背景を覆うため）
- 音声ファイル読込時・マイク入力時は対象がないため UI ごと非表示になる
- 録画（`Recorder`）は `canvas.captureStream()` を使うため無改修で動画合成込みの録画が可能。オフライン書き出しでの動画合成は §14.8.2（Phase 10.2）に定める

旧 Phase 4 で実装中止となっていた §8「スロット・再生キュー仕様」を、`doc/plan-phase12.md` に基づき再実装した。

- 再生モードで3スロット。各スロットへ独立にファイルを読込でき、番号/ファイル名/種別/アクティブ状態の表示と 選択/削除 操作を持つ
- 再生終了時に次の設定済みスロットへ自動循環（1スロットのみならループ再生）。次へ/前へボタンで手動循環も可能
- 既存のファイル選択・ドラッグ&ドロップはアクティブスロットへの読込として動作（互換維持）
- 録画モードは自動循環せず、録画中はスロット操作を受け付けない
- `AudioEngine.connectMedia()` は要素→ソースノードを `WeakMap` でキャッシュし、スロット再選択に対応

### Phase 13: モバイルレイアウト対応（実装済み）
狭幅画面（700px以下）での利用性を改善した。CSS のみの変更で、デスクトップレイアウトには影響しない。

- 縦積みレイアウト（上: ビジュアライザー 44dvh、下: 操作パネル全幅・ページスクロール）
- タッチターゲットの拡大（ボタン最小38px・セレクト36px・スライダー32px）と `touch-action: manipulation` によるダブルタップズーム遅延の無効化
- `100dvh` によるモバイルブラウザのアドレスバー変動への追従（非対応環境は `100vh` フォールバック）

### Phase 14.1: オフライン書き出しの動画デコード高速化（実装済み）
`doc/plan-phase14.md` に基づき、動画合成のフレーム取得をデコーダー方式優先に切り替えた。詳細は §14.8.2 に定める。

- `js/webm-demuxer.js` を新規追加（自前 WebM/EBML デマルチプレクサ。`js/webm-muxer.js` の読み取り側対応物）
- フレーム取得を `frameAt(tSec)` インターフェースで抽象化し、デコーダー方式（WebM + `VideoDecoder`）とシーク方式（既存）を自動選択・自動フォールバック
- 使用済み `VideoFrame` は即時 `close()` し、デコード先読みも供給キュー制限で抑えてメモリを管理する
- MP4 デマルチプレクサ（Phase 14.2）は `doc/plan-phase14.md` §4 に設計のみ記録し、将来対応とする（MP4 はシーク方式で動作する）

## 21. 受け入れ条件

以下を満たしたとき、初期仕様を達成したと判断する。

1. 音声ファイルまたは動画ファイルをブラウザで読み込める
2. 再生中にリアルタイムでアナライザーが表示される
3. アナライザータイプを複数から選択できる
4. 再生中に色相・彩度・輝度などを変更できる
5. 1〜4レイヤー構成を切り替えられる
6. 録画モードで単一スロットの録画ができる
7. 録画結果を保存できる
8. 背景が常に黒である
9. 表示比率として 16:9 / 1:1 を選択できる
10. 特別なライブラリ導入なし、または最小限で起動できる

### Phase 3 追加受け入れ条件
11. アナライザータイプとして棒グラフまたは円形放射を選択できる
12. 表現方法として棒 / 波形線 / 点を選択できる
13. 線の太さおよび密度を再生中に変更できる
14. 密度を最小側にしてもアナライザーが消失しない
15. 棒グラフでは通常 / ミラー上下 / ミラー左右を切り替えられる
16. 0dB基準位置に対するオフセットを最大 99 まで調整できる
17. 色相ランダマイズを実行できる
18. 再生中に色相が連続変化するモードを有効化できる
19. 残像表現のパラメーターを調整できる
20. 残像の過剰設定時に描画が飽和しない（上限クランプが機能する）

## 22. 制約・注意事項

- 対応メディア形式はブラウザ実装に依存する
- ブラウザの自動再生制限により、初回再生にはユーザー操作が必要
- `MediaRecorder` の出力形式はブラウザ差異がある
- `mp4` の安定出力は標準ブラウザ機能だけでは難しい
- 初版では高機能動画編集ではなく、**アナライザー録画** に目的を限定する

---

## 23. 今後の検討項目

初版で挙げた候補（プリセット保存・JSON入出力・動画合成・画質/FPS指定録画・レイヤーブレンド・マイク入力・フルスクリーン・ショートカット）は Phase 8〜11 で、3スロット再生キューは Phase 12、モバイルレイアウトは Phase 13、動画デコード高速化（WebM）は Phase 14.1 で実装済み。現時点の残候補:

- オフライン書き出しの MP4 デマルチプレクサ対応（Phase 14.2。設計は `doc/plan-phase14.md` §4。現状 MP4 はシーク方式で動作するため機能欠落はない）
- モバイルでの録画品質改善（`MediaRecorder` のブラウザ実装差に依存するため対応範囲は限定的）

## 24. 実装優先判断

本プロジェクトでは、以下を優先する。

1. **簡単に起動できること**
2. **リアルタイムで気持ちよく動くこと**
3. **視覚表現を柔軟に変えられること**
4. **録画できること**
5. **UIが直感的であること**

装飾的なUI改善は後回しとし、まずはコア機能の完成を優先する。

本仕様書は初期設計であり、実装の進行に応じて更新する。

## 1. テスト方針

アプリ本体は「ライブラリなし・ビルドなし」の方針のため、テストも同方針で構成する。

| レイヤー | 手段 | 対象 | 実行方法 |
|---------|------|------|---------|
| L1 単体テスト | Node 標準機能のみの自作ミニランナー | vis-utils / history-buffer / 粒子プール等の純ロジック | `node test/run-all.mjs` |
| L2 ビジュアルハーネス | ブラウザで開く自己診断ページ（モック音源） | 全レンダラーの実描画（例外・非空描画・FPS） | `test/visual-harness.html` を開く |
| L3 手動チェックリスト | 人手 | UI・録画・統合動作 | 本書 §5 のチェックリスト |
| L4 性能測定 | ハーネス内蔵の計測モード | フレーム時間 | 本書 §6 |

- テストコードは `test/` 配下に置き、アプリ実行には一切影響しない（index.html から参照しない）。
- テストコード内では `import/export`（ESM）・`Math.random()` シード化など、
  アプリ本体と異なる書式を許可する。

## 2. テスト環境・土台

### 2.1 ミニテストランナー（M1-1）

```text
/test
├─ run-all.mjs          # test/unit/*.mjs を順に import して実行、結果集計
├─ lib/tester.mjs       # test(name, fn) / assertEq / assertClose / assertThrows
├─ lib/mock-audio.js    # L2 用モック音源（ブラウザ用、classic script）
└─ unit/
   ├─ vis-utils.test.mjs
   ├─ history-buffer.test.mjs
   ├─ particles-pool.test.mjs
   └─ voronoi.test.mjs
```

- `tester.mjs` は約 50 行想定: 失敗時に期待値/実際値を表示し、プロセス終了コードで
  green/red を返す（PR 記載用）。
- アプリ本体は classic script のため、unit テストからは
  `vm.runInContext`（既存 `webm-duration` テストと同方式）で読み込む。
  読み込みヘルパー `loadScript(path, extraGlobals)` を `tester.mjs` に含める。

### 2.2 モック音源 `mock-audio.js`（M1-13）

AudioEngine 互換のインターフェースを持ち、**再現可能な合成データ**を返す。

| パターン | 内容 | 主な用途 |
|---------|------|---------|
| `silence` | 全ビン 0 / 波形 128 固定 | 無音時挙動（消滅・収束） |
| `sine(freqBin)` | 特定ビンのみ 255、波形は正弦波 | lissajous 形状・単一帯域反応 |
| `sweep` | ピークビンが時間で移動 | spectrogram の模様・帯域割当 |
| `pulse(bpm)` | 低域が周期的にバースト | ビート検出・ripple・particles |
| `noise(seed)` | シード付き疑似ランダム | 全タイプの一般動作・性能測定 |

- `mockEngine.setPattern(name)` / `captureFrame()` / `getLayerData()` /
  `getTimeDomainData()` を実装。時刻は外部から `tick(dtMs)` で進める（決定的）。

## 4. L2 ビジュアルハーネス設計（`test/visual-harness.html`）

### 4.1 目的

実ブラウザで全レンダラーを機械的に巡回し、「例外が出ない・何かが描かれる・速度が出る」を
半自動で確認する。実装者のセルフチェックと PR エビデンスに使う。

### 4.2 構成

- アプリ本体の `js/` を script タグで読み込み、AudioEngine だけ `mock-audio.js` に差し替える。
- 画面: キャンバス + 制御パネル（開始/停止・パターン選択・結果テーブル）。
- 自動巡回モード: 全ケース（§4.3）を各 2 秒描画 → 判定 → 次へ。

### 4.3 判定項目（1ケースごと）

| 判定 | 方法 | 合格基準 |
|------|------|---------|
| J1 例外なし | `window.onerror` + try/catch 計数 | エラー 0 件 |
| J2 非空描画 | 2秒目に `getImageData` を 16×16 グリッドでサンプル | 背景色以外のピクセル ≥ 1%（`noise` パターン時） |
| J3 無音消滅 | `silence` に切替 3 秒後に J2 と同サンプル | タイプ規定どおり（消滅系: 背景のみ / 収束系: 中央近傍のみ） |
| J4 FPS | `requestAnimationFrame` 間隔の平均 | §6 の基準 |
| J5 リサイズ耐性 | ケース中に canvas を 2 回リサイズ | J1 継続 |

### 4.4 巡回ケースと結果出力

- ケース = 全タイプ × 対応表現方法 × レイヤー数 {1, 4} × 背景 {黒, 白}
 （ケイパビリティマップから自動生成。おおよそ 100 ケース強）。
- 結果は画面表示に加え JSON でダウンロード可能にする:
  `{case, j1..j5, fps, elapsed}` の配列。PR に添付する（plan §5）。

### 4.5 既存タイプのピクセル同等性検証（M1-10 / M4-6 用）

- モード「回帰比較」: mock `noise(seed=42)` で 60 フレーム進めた時点の
  `canvas.toDataURL()` ハッシュ（SHA-256）を表示。
- 手順: M1 実装**前**の main で採取したハッシュ一覧（bar/radial × 3表現 × ミラー3種 ×
  レイヤー{1,4}）を `test/baseline-hashes.json` に保存 → M1 実装後・M4-6 実装後に同モードで
  一致確認。**mock 音源が決定的であることが前提**（§2.2）。
- 注意: 同一マシン・同一ブラウザバージョンでの比較に限る（レンダリング差異のため）。
  比較不能な場合は目視スクリーンショット比較に切り替えてよい（PR に明記）。

## 5. L3 手動チェックリスト

### 5.1 既存機能回帰（M1 完了ゲート・M6 最終）

- [ ] 音声ファイル読込→再生→bar/radial 各表現が Phase 5 と同様に表示される
- [ ] 色相・彩度・輝度・感度・スムージング・密度・オフセット・残像が全て効く
- [ ] 3種ランダマイズが動く
- [ ] 16:9/1:1・背景黒/白の切替
- [ ] 録画開始→停止→保存（MP4/WebM）→再生成功・A/V同期・Duration 正常

### 5.2 新タイプ個別（各マイルストーン PR 時）

各タイプで共通:
- [ ] 実音源（音楽ファイル）で 30 秒再生し、spec §6 の受け入れ条件を満たす
- [ ] 対応スライダー（history/motion/particles/physics のうち該当分）が効く
- [ ] 非対応 UI が非表示になっている
- [ ] タイプ切替往復（他タイプ→戻る）で状態が正しくリセットされる
- [ ] 残像強度 0 / 5 / 10 で破綻しない

タイプ固有の追加確認:
- [ ] T1/T2: 曲の展開（静→動）が模様の変化として視認できる
- [ ] T3: 手前の稜線が奥を隠す。`historySeconds` 1↔8 で奥行きが変わる
- [ ] T5/T6: 奥行きの明度差が背景白でも成立する
- [ ] T7: 静かな曲でまばら・激しい曲で多量。フリーズなし（粒子上限確認）
- [ ] T8: ビートのある曲で波が同期。アンビエント曲で過剰発生しない
- [ ] T10: ブロブの融合が見える。録画ファイルにも融合が写っている（filter 検証）
- [ ] T11: 音楽で形が常に変化。無音で中央収束
- [ ] T13: セルの脈動が帯域と対応（低音で左側が反応）

### 5.3 録画統合（M6-3）

- [ ] 全 15 タイプで各 10 秒録画 → 保存 → OS プレイヤーで再生成功
- [ ] WebM 保存分: 長さ表示が約 10 秒（Duration パッチ有効）
- [ ] 録画中にタイプ切替・ランダマイズを行っても録画が完走する
- [ ] 録画中 30fps 維持（rec-status 目視 + §6 計測）

### 5.4 ランダマイズ強度テスト（M6-1）

- [ ] アナライザーランダマイズを 100 回連打（再生中）し、例外・白画面・UI不整合が出ない
- [ ] 各回で選ばれた組み合わせがケイパビリティ表に適合（ハーネスの自動判定モードで代替可）

## 6. L4 性能テスト

### 6.1 測定方法

- ビジュアルハーネスの「性能モード」: `noise` パターン・1920×1080 キャンバス・
  各タイプ 10 秒描画し、フレーム時間の **平均 / p95** を記録。
- 測定機はミドルレンジ想定（例: 内蔵GPU のノートPC）。測定機スペックを結果に併記。

### 6.2 合格基準

| 条件 | 基準 |
|------|------|
| 通常描画（全タイプ） | 平均 ≤ 12ms かつ p95 ≤ 16.7ms |
| 4レイヤー時（対応タイプ） | p95 ≤ 16.7ms |
| 残像強度 10 併用 | p95 ≤ 16.7ms |
| 録画中（30fps 設定） | p95 ≤ 33ms |

- 未達の場合は spec §6 の性能予算（要素数）を下げて再測定し、確定値を仕様書へ反映する
 （plan §7 の方針どおり fps 優先）。

## 8. 不具合管理

- 発見した不具合は GitHub Issue に「タイプID + 現象 + 再現手順（ハーネスのケース名 or
  mock パターン）」で起票する。
- 再現可能な不具合は、可能な限り L1 単体テストまたはハーネスケースとして固定化してから
  修正する（回帰防止）。
